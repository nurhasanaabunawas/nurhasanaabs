    Evaluation Plan

    Kebanyakan sistem basis data menyediakan cara untuk melihat rencana evaluasi yang dipilih untuk menjalankan permintaan yang diberikan. Biasanya paling baik menggunakan GUI yang disediakan dengan sistem basisdata untuk melihat rencana evaluasi. Namun, jika Anda menggunakan antarmuka baris perintah, banyak variasi dukungan basis data dari sebuah perintah" explain ", yang menampilkan rencana eksekusi yang dipilih untuk query yang dispesifikasikan.Sintaks jelas bervariasi dengan basis data yang berbeda:



    PostgreSQL

    menggunakan sintaks yang ditampilkan di atas.



    Oracle

    menggunakan sintaks "explain plan for". Namun, perintah menyimpan rencana pada resultant plan yang disebut "plan_table", bukannya menampilkan. Query “select * from table(dbms xplan.display);” menampilkan plan yang tersimpan (stored).



    DB2

    mengikuti pendekatan yang sama terhadap Oracle, tapi memerlukan program db2exfmt untuk dieksekusi untuk menampilkan plan yang tersimpan.



    SQL

    Server SQL memerlukan perintah "set showplan_text on" untuk dieksekusi sebelum mengirimkan permintaan; kemudian, ketika sebuah query dimasukkan, ketimbang menjalankan query, evaluation plan yang ditampilkan.



    MySQL

    MySQL menggunakan sintats yang sama pada PostgreSQL yaitu "explain", tapi outputnya adalah sebuah tabel yang isinya tidak mudah dimengerti. Namun, mengeksekusi peringatan (show warnings) setelah perintah "explain" menampilkan evaluation plan dalam format yang lebih mudah dibaca.



    Perkiraan biaya untuk rencana juga ditampilkan bersama dengan rencana. Hal ini bernilai mencatat bahwa biaya biasanya tidak dalam setiap eksternal satuan berarti, Seperti detik atau operasi I/O, tetapi lebih pada unit apapun biaya model yang optimizer gunakan. Beberapa optimisasi seperti tampilan PostgreSQL dua jumlah biaya perkiraan; pertama menunjukkan biaya untuk outputting hasil pertama, dan kedua menunjukkan perkiraan biaya untuk outputting semua hasil.








    MySQL Evaluation Plan

    id

    menampilkan nomor urut dari query SELECT yang dimiliki baris ini. karena query tidak berisi subquery atau gabungan, maka id untuk kedua baris adalah 1

    Select_type

    berisi tentang apakah query berisi subqueri atau gabungan. Hasil EXPLAIN gambar disamping muncul "SIMPLE" yang berarti tidak ada subqueri / gabungan

    table

    menampilkan nama tabel yang digunakan dalam query

    type

    bagaimana tabel itu diakses. eq_ref merupakan tipe gabungan (join type)

    possible_keys

    menjelaskan jika tabel memiliki indeks untuk mencari baris, indeks dapat dipilih. Dan pada gambar disamping, dijelaskan bahwa keduanya memiliki indeks PRIMARY





    key_len

    panjang kunci yang digunakan MySQL, digambar disamping menjelaskan bahwa kedua baris memiliki 4 byte

    ref

    kolom yang dibandingkan dengan indeks yang diberi nama dikolom kunci

    rows

    menunjukkan berapa data yang diambil dalam tabel, tergantung dengan query apa yang dipakai

    extra

    menunjukkan informasi tambahan tentang query processing, dan pada gambar disamping pada kedua baris menunjukkan "Using Index"
